Building a Release

The main steps in building a release are as follows:
  • Create the directory to contain the release. E.g. in a working copy of the pages project, svn add directory dist/working/110.n, where n is the release number. Then commit the change, and update the web-site copy of the pages project at /stages/web_static/smlnj/htdocs.
  • Write 110.n-README.txt and 110.n-README.html files and commit them. Also update NOTES/HISTORY file.
  • Run the allcross script in base/system to build boot directories for all the architectures and create tarballs from them. Copy these tarballs to the dist/working/110.n directory created in the first step.
  • Export source directories and create tarballs and transfer them to the distribution directory.
  • Create .dmg file for Mac OS X installation, and a Windows .zip file. (Details?). Add these to the distribution directory.
  • Add a link to dist/working/110.n/110.n-README.html to dist/index.html.
Following are specific instructions for building a SML/NJ release, divided by operating system (Linux, OS X, Windows).

Building on UNIX

  1. Update current copy, or check out a fresh copy:
    • $ ./admin/refresh-all.sh
    • Or from a CS machine: svn co --username yourusernamehere https://smlnj-gforge/svn/smlnj/admin, then ./admin/checkout-all.sh.
  2. Compile to fixed point:
    • $ cd base/system
    • $ ./fixpt
  3. Take the .bin and .boot files and make them the official ones.
    • This depends on how many iterations it took to reach a fixed point. For example, if fixpt took two iterations on a x86 UNIX platform (the postfix notation follows a CPU-OS convention), then you'd do the following:
      • $ rm -rf sml.bin.* sml.boot.*
      • $ mv sml2.bin.x86-unix sml.bin.x86-unix
      • $ mv sml2.boot.x86-unix sml.boot.x86-unix
  4. Generate new library and heap files.
    • $ ./makeml
  5. Test the build.
    • $ ./testml
  6. Remove the old libraries.
    • $ rm -rf ../../lib/*
    • $ rm -rf ../../bin/.heap/*
    • $ ./installml
    • (To check everything - make sure everything is enabled in config/targets.)
  7. Build a working installation.
    • $ cd ../..
    • $ ./config/install.sh
  8. Edit the version number.
    • $ $EDITOR ./config/version
  9. Build to another fixed point in base/system.
    • $ cd base/system
    • $ ./fixpt
  10. Repeat steps above, starting at step after fixpt , and ending after running config/install.sh.
    • Can omit step involving testml. (Which seems broken in my notes anyway.)
  11. Cross compile.
    • $ cd base/system
    • $ ./allcross
  12. Edit/update the history file.
    • .../base/NOTES/HISTORY: Add a new log entry similar to others.
    • .../base/READMES/: Copy previous READ and edit; older versions give examples of format expectations.
      • Cut and paste relevant entries in HISTORY and format them.
      • Also check the SML/NJ Library's change log (smlnj-lib/CHANGES), since those changes often do not make it into the HISTORY file.
  13. Update web pages.
    1. Go to the relevant SML/NJ web server directory.
      • On the UChicago CS cluster at: /stage/web_static/smlnj/htdocs/dist/working
        • Ideally this all under Subversion control, but this convention isn't being followed 100%.
    2. Add directory for 110.XX.
    3. Copy previous index file over and update links.
    4. Build tarballs:
      • $ for i in alpha32-unix ...; do f=sml.boot.$i; t=boot.$i.tgz; tar cfz $t $f/*; done
    5. Copy tarballs and readme notes into release directory.
  14. Tag the release.
    • $ ./admin/make-release.sh
      Note that each major directory under smlnj has its own trunk and release branches, e.g. $smlnj/sml/trunk, $smlnj/sml/releases/release-110.74. ml-lpt does not have release branches.
  15. Create source tarballs.
    • $ ./admin/build-tar-files.sh
  16. Create DMG files for Mac.
    • (See below...)
  17. Create MSI files for Windows.
    • (See further below...)
  18. Sanity checks.
    • Download, install, and run tests.
  19. Update links.
    • Symbolic link at /stage/web_static/smlnj/htdocs/dist/working/current
    • Table at /stage/web_static/smlnj/htdocs/dist/working/index.html
    • What's new and links in /stage/web_static/smlnj/htdocs/index.html

Building a DMG

  1. Export the SML/NJ repository.
    • Example: admin/checkout-all.sh --export from some directory named after the new version.
  2. Enable building all targets.
    1. $ cp config/targets config/targets.customized
    2. $ vi config/targets.customized
  3. Note that if you are building on Snow Leopard, then you have to modify the makefile to guarantee that the runtime will work on Leopard. The following linker flags are neede when linking the various static libraries that make up the runtime system:
    -isysroot /Developer/SDKs/MacOSX10.5.sdk -mmacosx-version-min=10.5
  4. Build the exported copy and remove the boot file.
    1. Plain-old "$ config/install.sh"
    2. $ rm boot.x86-unix.tgz
  5. Create or modify a PackageMaker package.
    • Example: smlnj-110.68.pmdoc
      • I'm currently keeping this in ~/ML on the iMac.
    • The background image can be found inside earlier distributions.
  6. Build and test the .mpkg file.
  7. Create the DMG file using Disk Utility application.
    • Specifically, use the "File > New > Disk Image from Folder" command.
  8. Test the DMG file.
  9. Post online.
  10. Redo it all again, but remotely on a PPC system (currently advised to use g5.cs.uchicago.edu. -Note: As of 6/29/2010, I (jriehl) am unable to access g5. I advise checking with techstaff.)
    • Somewhere in there, there is a "move the built source tree to your desktop" step.

References: Apple's "Software Delivery Guide" (should be available from ADC, or inside of Xcode).
